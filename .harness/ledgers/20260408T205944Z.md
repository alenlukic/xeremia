# Run Ledger

## Outcome
- Task: Fold canonical `agentic-harness-template/` into the repo's live `.harness/` structure.
- Result: Canonical template content was adopted across the live harness, with repo-specific behavior intentionally preserved where blind replacement would have regressed local capabilities.
- Scope: Harness structure, top-level guidance alignment, canonical naming cleanup, and ledger index/schema consistency.

## Key Decisions
- Decision: Adopt template agent, command, and documentation files as the default source of truth.
  - Why: The template represented the clean canonical state the live harness was meant to converge on.
  - Tradeoff: Fast alignment with canonical structure reduced drift, but required careful exceptions for repo-local behavior.
- Decision: Merge `pipeline.py` instead of copying the template version wholesale.
  - Why: The template added important harness capabilities, but the repo version still carried richer evaluation and ledger-publishing behavior worth keeping.
  - Tradeoff: Merge work was more complex than a straight sync, but avoided losing repo-specific enhancements.
- Decision: Merge `pipeline.yaml` instead of copying the template version wholesale.
  - Why: Command registry, repo type, and `forbid_paths` are repo-specific and could not be safely normalized from the template.
  - Tradeoff: Canonical structure was preserved, but local config remained a deliberate maintenance surface.
- Decision: Keep the existing rich ledger `INDEX.json` schema and rewrite `rebuild_ledger_index` to match `publish_ledger`.
  - Why: The repo already depended on richer ledger metadata than the template schema provided.
  - Tradeoff: The repo stays slightly divergent from the template here, but avoids internal schema mismatch.

## Verification Learnings
- Migration verification should explicitly cover merged logic, not just copied files; `pipeline.py` evaluation behavior and ledger index rebuilding were both blind spots after the sync.
- Template fold-ins can appear complete while still hiding repo-specific regressions in configuration and post-processing paths.
- Final verification outcome remained acceptable for the run: QA `PASS`, Review `APPROVE`, Build `PASS`, Eval `PASS` (`83/B`), Regression `LOW`, Breaker `CONCERNS` with no blockers.

## Product / Stakeholder Learnings
- Canonical harness cleanup has value beyond aesthetics: reducing deprecated names and compatibility shims makes operator guidance easier to trust and lowers future onboarding confusion.

## Technical / Architecture Learnings
- Template syncs should treat `pipeline.py` and `pipeline.yaml` as merge-prone integration points, not pure copy targets.
- Ledger publication and ledger index rebuild paths must emit the same schema; allowing them to drift creates avoidable downstream inconsistency.
- Template-added capabilities such as state-machine, schedules, contract management, and bad-state scanning can be incorporated safely when the template is used as the base and repo enhancements are layered back in.

## Durable Repo Guidance
- Scope: repo-wide
  - Guidance: During template syncs, always diff and reconcile `pipeline.yaml` `commands`, `repo.type`, and `forbid_paths` as repo-specific configuration.
- Scope: repo-wide
  - Guidance: Treat `pipeline.py` as a merge surface whenever the repo carries custom evaluation, publishing, or scoring behavior.
- Scope: repo-wide
  - Guidance: Keep ledger schemas aligned between `publish_ledger` and `rebuild_ledger_index`; validate both together after any migration.
