# Run Ledger

## Outcome
- Task: Close scoring-weight persistence coverage gaps around `_ensure_table` and stale-key re-persist proof.
- Result: The follow-on closed both breaker `IMPORTANT` findings with a minimal production seam plus focused tests; review, QA, breaker, regression, and evaluation all cleared, while branch-wide build status remained conditional only because of two inherited full-suite failures.
- Scope: Limited to `src/harmonic_mixing/weight_service.py` and `src/tests/test_weight_service.py`, with no expansion into deferred `WATCH-*` items or unrelated persistence behavior.

## Key Decisions
- Decision: Extract `_ensure_table()` as a small callable seam at the top of `_load_from_db()` instead of broadening the persistence flow.
  - Why: This gave the test suite a precise hook to prove init-time invocation and to verify that an ensure-table failure does not block loading an existing persisted row.
  - Tradeoff: The unit coverage proves call-chain wiring and failure isolation, but not the real SQLAlchemy DDL body; verifying the actual table-create call remains an integration concern.
- Decision: Harden stale-key migration coverage by asserting on the state observed at `_persist_to_db()` time, not just on in-memory redistribution.
  - Why: This closes the parent breaker gap where stale-key cleanup could appear correct in memory while silently skipping re-persistence of the cleaned payload.
  - Tradeoff: The test is coupled to the current payload assembly shape, but that coupling is acceptable here because the persistence method serializes the same combined weight dict and round-trip persistence behavior is already covered elsewhere.

## Verification Learnings
- Targeted acceptance evidence can be strong enough to close a narrow follow-on even when the branch is not at a clean full-suite baseline, but the inherited failure lineage must be recorded explicitly so the scoped change is not blamed for unrelated red tests.
- A build-intent command chained with `&&` can leave a verification blind spot: if pytest fails first, the run has no lint result even when the scoped change itself is green.
- In this subsystem, `_ensure_table` invocation and graceful degradation were the right unit-test targets; verifying the actual DDL side effect belongs at an integration layer if stronger persistence assurance is later needed.

## Product / Stakeholder Learnings
- No new product or customer-facing behavior was introduced; the durable value of this run is reliability hardening in persistence coverage rather than a workflow change.

## Durable Repo Guidance
- Scope: subsystem-specific
  - Guidance: When persistence setup and row loading have different failure modes, separate the setup step behind a tiny seam and keep its failure non-fatal if persisted data can still be read; this makes resilience testable without broad production churn.
- Scope: subsystem-specific
  - Guidance: For legacy-key cleanup paths, tests should prove that sanitized state is re-persisted, not only that runtime weights look correct after load.
- Scope: repo-wide
  - Guidance: When a scoped follow-on ships on top of known branch-lineage failures, preserve the narrow PASS evidence and classify branch health separately instead of broadening the fix scope opportunistically.
