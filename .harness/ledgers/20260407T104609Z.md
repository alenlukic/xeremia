- Decision: Keep the failure framed as an in-scope correctness problem rather than an out-of-scope regression.
  - Why: Regression analysis found no concrete adjacent drift; the observed failure was the feature itself not working in the live UI.
  - Tradeoff: Avoided broadening into unrelated layout/search/build work, while still recording the blocker clearly.

## Verification Learnings
- Passing focused component tests and even full client Vitest coverage were insufficient here; the gap was integrated runtime behavior under the live app.
- Save-state logic inside a React state-updater is a high-risk pattern for this repo's React StrictMode setup because it can double-run timers and status transitions while still looking correct in tests.
- Version guards must wrap all save-success side effects, not only some state writes; stale success completions can otherwise clear current errors or fire outdated success callbacks.
- Repo-wide build evidence can still fail a client-only run on pre-existing Python structure checks. That blocks build verification, but should be recorded separately from task-fit failures and regression claims.

## Product / Stakeholder Learnings
- Local, subtle status UX only works if the save signal is visible in the real interaction path; "technically rendered under some props" is not enough for acceptance.
- Combining long warning copy and transient save feedback in the same local region increases the chance that the warning path appears healthy while save-state feedback remains effectively absent.

## Durable Repo Guidance
- Scope: subsystem-specific
  - Guidance: For `useWeights` and similar client hooks, keep state updaters pure and move debounced persistence side effects outside updater callbacks, especially under React StrictMode.
- Scope: subsystem-specific
  - Guidance: When a feature depends on transient UI states (`Saving`, `Saved`, error banners), require at least one integrated assertion that exercises real async flow instead of only prop-level rendering or fake-timer-heavy tests.
- Scope: repo-wide
  - Guidance: When breaker finds a concrete live blocker after QA/build/eval, preserve it as a follow-on contract/run rather than folding more churn into the failed run.
