# Harness Glossary

## contract

A durable implementation or follow-on spec intended to be picked up by a later delivery run.

Canonical reference: `.harness/workspace/contracts/README.md:1`

## control plane

The runtime, schedules, state machine, rules, and pipeline configuration that keep the harness operating.

Canonical reference: `.harness/control/pipeline.yaml:1`

## history

Durable historical exhaust: runs, ledgers, and PR descriptions.

Canonical reference: `.harness/history/ledgers/README.md:1`

## inbox

The primary human-facing surface for seeing what changed, what is blocked, and what needs attention.

Canonical reference: `.harness/workspace/inbox/README.md:1`

## intake

The canonical landing zone for ongoing context ingest and routing.

Canonical reference: `.harness/intake/INDEX.md:1`

## knowledge

Derived docs and memory that help agents orient without rereading all history.

Canonical reference: `.harness/knowledge/memory/README.md:1`

## ledger

A compact, durable record of a completed run's most important decisions, learnings, and outcomes.

Canonical reference: `.harness/history/ledgers/README.md:1`

## spec layer

Static agent, command, and rule definitions. These define behavior but do not represent live work.

Canonical reference: `.harness/spec/agents/INDEX.md:1`

## work index

The categorized registry of outstanding work and follow-on items.

Canonical reference: `.harness/workspace/work/README.md:1`

## workspace

Active operator-facing surfaces: inbox, contracts, product-feedback state, and open work.

Canonical reference: `.harness/workspace/inbox/README.md:1`
