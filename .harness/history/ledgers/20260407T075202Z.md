# Run Ledger

## Outcome
- Task: Run a product-feedback pass across dj-tools and convert the strongest findings into a focused development contract.
- Result: The run converged on missing artist visibility in browse and match results as the highest-confidence, smallest coherent delivery slice, and produced a narrow contract to repair that API-to-UI path.
- Scope: Product feedback, customer/domain critique, and contract scoping only; no implementation work was performed in this run.

## Key Decisions
- Decision: Scope the follow-on contract to artist names specifically.
  - Why: Both design and SME review independently identified missing artist metadata as a P0 defect on the core DJ workflow, and it was the smallest overlapping fix that materially improves track recognition.
  - Tradeoff: Broader workflow and trust issues such as export, transition chaining, error surfacing, trait hydration, and weight/config cleanup were explicitly deferred to avoid bundling unrelated work.
- Decision: Keep the contract focused on aligning the existing backend-to-frontend data path rather than redesigning the product surface.
  - Why: The strongest evidence pointed to an incomplete runtime contract, not a missing feature category; using the current track-to-artist model keeps the fix narrow and verifiable.
  - Tradeoff: The contract does not address larger roadmap gaps even though some of them may have equal long-term product value.

## Verification Learnings
- Fusion weights are a trust-breaking blind spot: the UI and API expose `FUSION_*` controls, but the scoring path does not use them. Future verification should require proof that every exposed control changes runtime behavior or be removed from the product surface. Source: `SME_RECOMMENDATIONS.md` P1-2.
- Weight configuration drift is another trust-boundary failure: documented `.env.example` factors do not match the implemented scoring factors, and the documented totals are inaccurate while runtime normalization hides the mismatch. Config examples should be checked as executable contracts, not treated as documentation-only. Source: `SME_RECOMMENDATIONS.md` P0-3.
- Match-detail trait data and API error handling remain verification-sensitive areas because the current product can mask backend failures as normal empty states or rely on cache timing for trait display. These were not selected for the first delivery slice, but they remain high-signal checks for future runs. Source: `DESIGN_RECOMMENDATIONS.md` P0-2 and P0-3.

## Product / Stakeholder Learnings
- Artist names are not cosmetic metadata for DJs; they are primary identity. Search suggestions already prove the product knows this, so losing artists in browse and matches creates a jarring workflow break at the moment of decision.
- The most important product gap beyond the chosen contract is workflow closure: dj-tools can analyze "what should I play next?" but still lacks a durable path back into performance software or a native set-building loop. Source: `SME_RECOMMENDATIONS.md` P0-2, P1-1, and P1-4.
- Trust in the scoring model depends on visible alignment between labels, controls, and runtime behavior. Inert weights, misleading config, or unclear factor provenance weaken the product's core differentiator more than minor UI polish does.

## Durable Repo Guidance
- Scope: repo-wide
  - Guidance: When multiple feedback reports overlap, prefer the smallest shared P0 that repairs a broken core workflow over bundling adjacent improvements into one contract.
- Scope: subsystem-specific
  - Guidance: Treat API schema fields and config examples as runtime contracts. If a field such as `artist_names` or a documented weight factor exists in the schema or sample config, verification should confirm it is actually populated and used.
- Scope: repo-wide
  - Guidance: For scoring products, every user-visible factor, tooltip, and control must map to real behavior. If the system cannot prove that link, remove the surface or mark it explicitly as deferred.

- Clean up phantom config factors and make default/example weights sum exactly to the documented total. Source: `SME_RECOMMENDATIONS.md` P0-3.
- Add an export path and transition chaining so analysis can feed real DJ set-building workflows instead of ending at inspection. Source: `SME_RECOMMENDATIONS.md` P0-2 and P1-1.
